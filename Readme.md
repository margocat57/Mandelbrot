# Изучение SIMD-функций на примере алгоритма построения множества Мандельброта

В данной работе исследуются различные способы оптимизации одного из вычислительно затратных алгоритмов — построение множества Мандельброта. Основное внимание уделяется как оптимизациям, автоматически выполняемыми компиляторами g++ и clang++, так и ручным преобразованиям кода, направленным на ускорение вычислений за счёт использования SIMD-инструкций.

## Рассматриваемый алгоритм

Построение множества Мандельброта происходит следующим образом:

1. Выбирается область на комплексной плоскости, которую хотим визуализировать.

2. Для каждой точки  \( z_0 = (x_0, y_0) \)  из этой области на каждом шаге координаты пересчитываются по формулам:

```
\[
\begin{cases}
x_n = x_{n-1}^2 - y_{n-1}^2 + x_0 \\
y_n = 2 x_{n-1} y_{n-1} + y_0
\end{cases}
\]
```

3. Итерации продолжаются до тех пор, пока модуль значения \( z_n \)  не превышает заданный порог \( R_max\) или не достигается ограничение на число шагов.

4. Цвет точки \( z_0 \) определяется тем, сколько итераций понадобилось, чтобы модуль значения \( z_n \) превысил \( R_max\) (или тем, что он не превысил вообще).

## План работы

В работе рассматриваются три реализации алгоритма:

1. Последовательная обработка точек без использования векторизации.

2. Реализация с использованием массивов - обработка ведётся блоками по 4 точки, что позволяет компилятору распознать возможность векторизации.

3. Применяются явные SIMD-инструкции для параллельной обработки нескольких точек.

## Методика тестирования

Для измерения времени выполнения используется функция `clock_gettime` с таймером `CLOCK_MONOTONIC_RAW`.

Данная функция возвращает время, считываемое напрямую с аппаратного таймера и не подверженное корректировкам со стороны операционной системы. В частности, это исключает влияние любых механизмов коррекции времени со стороны ОС: как скачкообразных (`NTP`), так и плавных (`adjtime`), а также внутренних поправок ядра, применяемых к `CLOCK_MONOTONIC`.

Используется следующая схема замеров:

- В рамках одного теста выполняется 100 последовательных запусков алгоритма, и время замеряется по ним.
- Каждая реализация тестируется 40 раз:
  - первые 10 запусков используются как разогревочные, они необходимы для прогрева кэшей и стабилизации частоты процессора;
  - оставшиеся 30 запусков используются для анализа.
- Для каждой реализации проводится 3 независимых серии измерений.


Код компилируется с флагами:

```
-march=native -Wall -Wextra
```

- `-march=native` позволяет использовать все доступные инструкции процессора (включая SIMD);
- `-Wall -Wextra` помогают избежать ошибки реализации, вызывающие неопределенное поведение

Эксперименты проводились на процессоре `Intel Core Ultra 9 285H`, операционная система `Linux 6.18.18-1-MANJARO`

Выполнение закреплялось за конкретным ядром `taskset -c 3 ./program` и ядро изолировалось от планировщика ОС `GRUB_CMDLINE_LINUX_DEFAULT="isolcpus=3"`

Также в процессе замеров контролировалось, что тактовая частота процессора фиксирована и отсутствует тротлинг.

## Сравнение различных реализаций алгоритма

Каждая реализация компилировалась с использованием компиляторов g++ и clang++ с флагами оптимизации -O2 и -O3. Данные уровни оптимизации выбраны, так как являются наиболее распространенными на практике.

### Обоснование необходимости отключения отрисовки

Покажем, что отрисовка множества Мандельброта вносит статистически значимый вклад в общее время выполнения программы. Для этого после каждого теста дополнительно вызывается функция отрисовки.

Нулевая гипотеза - распределения времени выполнения с отрисовкой и без неё совпадают, то есть различия между измерениями отсутствуют. Уровень значимости принят равным \( \alpha = 0.05 \). Для проверки гипотезы используется U-критерий Манна–Уитни.

| Реализация\Компилятор\Оптимизация | p-value |
|-----------|-----------|
| Naive g++ -O2  | $5 \cdot 10^{-31}$  |
| Naive clang++ -O2       |  $3 \cdot 10^{-30}$       |
| Naive g++ -O3       |   $9 \cdot 10^{-30}$     |
| Naive clang++ -O3       |  $10 \cdot 10^{-31}$        |
| Array g++ -O2  | $5 \cdot 10^{-31}$  |
| Array clang++ -O2       | $5 \cdot 10^{-31}$        |
| Array g++ -O3       |    $6 \cdot 10^{-31}$       |
| Array clang++ -O3       | $6 \cdot 10^{-30}$        |
| SIMD g++ -O2  | $5 \cdot 10^{-31}$  |
| SIMD clang++ -O2       |   $5 \cdot 10^{-31}$       |
| SIMD g++ -O3       |   $7 \cdot 10^{-31}$      |
| SIMD clang++ -O3       |   $1 \cdot 10^{-31}$       |

Во всех случаях нулевая гипотеза отвергается, что свидетельствует о статистически значимом влиянии отрисовки на время выполнения.

Поскольку целью эксперимента является сравнение эффективности различных реализаций алгоритма, при проведении измерений отрисовку множества Мандельброта следует отключать.

#### Сравнение ассемлерного кода, сгенерированного компилятором g++ при флагах -О2 и -О3

## Выводы

1. Компилятор не всегда способен самостоятельно получить максимально эффективный код, даже при высоких уровнях оптимизации. Даже если переписать алгоритм в виде обработки массивов по 4 элемента, то все равно часть действий, которые можно векторизовать вручную, компилятор оставляет в скалярном виде

2. В теории SIMD-версия, которая считает сразу 4 точки, могла бы ускорить программу почти в 4 раза. На практике до такого ускорения дойти не получается из-за устройства самого алгоритма.

   В одном ymm-регистре одновременно лежат 4 точки. Проблема в том, что они могут выйти за пределы области за разное число итераций. Например, три точки уже вышли за пределы `Rmax`, а четвёртая ещё нет. В обычной скалярной версии для первых трёх точек вычисления бы уже остановились, но в SIMD-варианте вычисления продолжаются для всего ymm-регистра.

   В коде это место выглядит так:

   ```cpp
   __m256d cmp2_d = _mm256_cmp_pd(R2, vRmax, _CMP_LE_OS);
   N = _mm256_sub_epi64(N, _mm256_castpd_si256(cmp2_d));

   int need_exit = _mm256_testz_pd(cmp2_d, cmp2_d);
   if (need_exit) break;
   ```

   Маска `cmp2_d` показывает, какие точки ещё остаются внутри допустимой области. Выйти из цикла можно только когда эта маска стала нулевой, то есть когда когда вычисления для всех четырёх точек завершились.

   Из-за этого SIMD-реализация всё равно даёт большой выигрыш, но фактическое ускорение оказывается меньше теоретического максимума в 4 раза.

3. Для g++ при переходе от -O2 к -O3 в ряде случаев наблюдается ухудшение производительности, связанное с отказом от векторизации в пользу скалярных преобразований с дополнительными обращениями к памяти.

4. В рассматриваемых тестах компилятор clang++ показывает лучшую производительность по сравнению с g++. Это связано с более агрессивной авто-векторизацией, развёрткой циклов и более эффективным использованием регистров c меньшим числом обращений к памяти.
